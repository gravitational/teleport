# Terraform provider architecture

This document describes the internal structure of the Teleport Terraform provider. For task-oriented instructions on
adding or converting resources, see [CONTRIBUTING.md](./CONTRIBUTING.md).

## Goals

The provider manages Teleport resources through the Terraform Plugin Framework. The current architecture separates
reusable Terraform lifecycle mechanics from resource-specific Teleport API calls so that new resources can be added with
a small amount of handwritten code.

The main goals are:

- keep Terraform lifecycle behavior consistent across resources;
- use generated Terraform schemas and conversion functions from `integrations/terraform/tfschema`;
- isolate Teleport API details from Terraform framework details;
- allow legacy generated resources and new generic-driver resources to coexist during migration.

## Package layout

The provider code lives under `integrations/terraform/provider`.

### `provider`

`provider/provider.go` is the Terraform provider entry point. It owns provider configuration, creates the authenticated
Teleport API client, exposes retry settings, and registers resources and data sources.

Resource registration is split into two groups:

- `internal/legacy`: existing generated resource and data source implementations.
- `internal/resources`: resources implemented with the generic driver.

`GetResources` and `GetDataSources` start with the legacy registry and then insert generic-driver resources. This lets
migrated resources replace legacy registrations without changing the provider's public Terraform resource names.

### `internal/tfdriver`

`internal/tfdriver` contains reusable Terraform resource and data source drivers.

Important types:

- `ResourceType[T, I]`: describes one Terraform resource for Teleport resource type `T` and identifier type `I`.
- `DataSourceType[T, I]`: describes one Terraform data source.
- `ResourceClient[T, I]`: Teleport API operations required by a managed resource: `Get`, `Create`, `Upsert`, and
  `Delete`.
- `DataSourceClient[T, I]`: Teleport API operation required by a data source: `Get`.
- `ResourceCodec[T]` and `DataSourceCodec[T]`: convert between Terraform objects and Teleport resource values.
- `IdentifierPolicy[T, I]`: extracts identifiers from Terraform state, imported IDs, and Teleport resource values.
- `ResourceNormalizer[T]`: applies resource-specific defaults or invariants before create and update.

The driver owns common lifecycle behavior:

- create preflight checks that prevent creating a resource that already exists in Teleport;
- create/update normalization before API calls;
- create/update retry loops for eventually consistent reads;
- update polling until the resource revision changes;
- read behavior that removes missing remote resources from state;
- delete behavior;
- import-state parsing and state population;
- consistent diagnostic wrapping through `internal/tfdiag`.

A resource descriptor supplies only the parts that differ by resource: schema/conversion functions, CRUD client, kind,
identifier policy, optional normalizers, and revision extraction.

### `internal/resources`

`internal/resources` contains resource descriptors. These files bind together:

- generated schema and conversion functions from `integrations/terraform/tfschema`;
- API clients from `internal/teleport`;
- identifier policy and import ID format;
- normalizers such as `tfdriver.CheckAndSetDefaults` or `tfdriver.ForceKind`;
- revision extraction for update convergence.

### `internal/teleport`

`internal/teleport` adapts `*client.Client` methods to `tfdriver` interfaces. These clients should be thin wrappers
around Teleport API calls. They should not know about Terraform plans, state, schemas, or diagnostics.

For example, an API adapter implements methods such as:

```go
func (c AppClient) Get(ctx context.Context, id tfdriver.NameIdentifier) (*types.AppV3, error)
func (c AppClient) Create(ctx context.Context, app *types.AppV3) error
func (c AppClient) Upsert(ctx context.Context, app *types.AppV3) error
func (c AppClient) Delete(ctx context.Context, id tfdriver.NameIdentifier) error
```

### `internal/legacy`

`internal/legacy` contains generated resource and data source implementations that predate the generic driver. It also
contains the legacy registry in `registry.go`.

Legacy files are generated by `go run ./gen/main.go`, normally through `make gen-tfschema`. The templates are in
`integrations/terraform/gen/*.go.tpl`. Legacy implementations should continue to work while individual resources are
migrated to `internal/resources` and `internal/teleport`.

### `tfschema`

`integrations/terraform/tfschema` contains generated Terraform schemas and generated copy functions for Teleport API
types. These generated functions are used by both legacy resources and generic-driver resources.

When protobuf schema changes affect Terraform, update the relevant `protoc-gen-terraform-*.yaml` file and run `make
gen-tfschema` from `integrations/terraform`.

## Identifiers

An identifier represents a unique resource in Teleport. This is how Terraform maps a TF resource to a Teleport resource stored in the cluster.
`internal/tfdriver/identifier.go` contains common identifier formats:

- `NameIdentifier`: Typical for most Teleport resources that are uniquely identifier by `metadata.Name`.
- `ScopeQualifiedNameIdentifier`: Typical for scoped Teleport resources that are uniquely identified by `(name, scope)`.
- `CompositeIdentifier`: An identifier distinguished by two parts, i.e. an Access List Member  is identified by the
                         name of the Access List _and_ the name of the member.
- `SingletonIdentifier`: Teleport singletons are identfied by a fixed identifier.

## Normalization and defaults

Use normalizers to enforce resource invariants before calling Teleport APIs:

- `tfdriver.CheckAndSetDefaults[T]()` calls the resource's `CheckAndSetDefaults` method.
- `tfdriver.ForceKind[T](kind)` sets the Teleport resource kind when Terraform should not require users to provide it.
- `tfdriver.ResourceNormalizers[T]` composes multiple normalizers.

Prefer normalizers over duplicating defaulting logic in API adapters.
