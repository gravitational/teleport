Beams make excellent remote development environments — isolated microVMs with
full access to Teleport infrastructure and inference endpoints. This guide
explains how to configure SSH access to a Beam and connect your editor to it.

## Prerequisites

- `tsh` version 18.8.0 or newer. See [Download Client Tools](https://goteleport.com/download/client-tools/).

- An active Teleport session (`tsh login`).

- An active Beam. Run `tsh beams add --no-console` to create one without
  immediately opening a shell:

  ```code
  $ tsh beams add --no-console
  ◆ created <beam-name>
  ```

## Configure SSH access

These two steps are required before connecting any editor to a Beam.

### Generate your SSH configuration

Run `tsh config` to append the SSH proxy configuration for your cluster to
`~/.ssh/config`:

```code
$ tsh config >> ~/.ssh/config
```

This adds a wildcard entry that matches all nodes in your cluster.

### Update the ProxyCommand for Beams

Beams are addressed by alias (`teleport.internal/beams/alias=<name>`) rather
than by hostname. The default ProxyCommand generated by `tsh config` won't
work as-is. Update the wildcard entry for your Beams cluster:

1. Remove the `Port 3022` line.
2. Add `StrictHostKeyChecking no` (Beam host keys rotate with each new VM).
3. Replace the `ProxyCommand` so it extracts the Beam name from the hostname:

```diff
Host *.<tenant-name>.beams.run !<tenant-name>.beams.run
-    Port 3022
-    ProxyCommand "/path/to/tsh" proxy ssh --cluster=<tenant-name>.beams.run --proxy=<tenant-name>.beams.run:443 %r@%h:%p
+    StrictHostKeyChecking no
+    ProxyCommand sh -c '"/path/to/tsh" proxy ssh --cluster=<tenant-name>.beams.run --proxy=<tenant-name>.beams.run:443 %r@teleport.internal/beams/alias=$(echo %h | cut -d. -f1)'
```

Replace `/path/to/tsh` with the path returned by `which tsh`. On macOS with
Teleport Connect, the path is
`/Applications/Teleport Connect.app/Contents/MacOS/tsh.app/Contents/MacOS/tsh`.

Verify the connection:

```code
$ ssh beams@<beam-name>.<tenant-name>.beams.run
beams@<beam-name>:~$
```

## VS Code

Connect VS Code to a Beam using the [Remote - SSH
extension](https://marketplace.visualstudio.com/items?itemName=ms-vscode-remote.remote-ssh).

### Add an SSH host entry

Add a dedicated host entry for the Beam in `~/.ssh/config`. The `vscode+`
prefix gives it a distinct name in VS Code's host picker:

```
Host vscode+<beam-name>.<tenant-name>.beams.run
    HostName <beam-name>.<tenant-name>.beams.run
    User beams
    UserKnownHostsFile /dev/null
    RemoteCommand bash
```

The `HostName` directive causes the wildcard entry from the previous section to
apply, so the ProxyCommand runs automatically. The `RemoteCommand bash` line
works around a TTY compatibility issue between VS Code's SSH client and the
Beam shell.

### Enable RemoteCommand

VS Code must be configured to forward the `RemoteCommand` directive. Open
settings and add:

```json
"remote.SSH.enableRemoteCommand": true
```

### Connect

1. Open the command palette (`Cmd+Shift+P` / `Ctrl+Shift+P`).
2. Select **Remote-SSH: Connect to Host…**
3. Choose `vscode+<beam-name>.<tenant-name>.beams.run`.

Upon the initial connection, VS Code will install its server component on the Beam and open a remote workspace.

<Admonition type="tip">
- Official AI extensions such as [Claude Code](https://marketplace.visualstudio.com/items?itemName=anthropic.claude-code) and [ChatGPT Codex](https://marketplace.visualstudio.com/items?itemName=openai.chatgpt) work out of the box.
- Official GitHub extensions
  ([GitHub Pull Requests](https://marketplace.visualstudio.com/items?itemName=GitHub.vscode-pull-request-github),
  [GitHub Actions](https://marketplace.visualstudio.com/items?itemName=GitHub.vscode-github-actions)) can help with GitHub authentication.
</Admonition>

<Admonition type="note">
Beam certificates rotate when you re-authenticate with `tsh login`. If VS Code
shows a connection error after the session expires, log in again with `tsh
login` and reconnect.
</Admonition>

### Automated setup with Ansible

<Admonition type="tip">
Before running this playbook, complete the [Configure SSH
access](#configure-ssh-access) section above. The playbook connects to the Beam
over SSH and relies on the modified ProxyCommand already being in place in
`~/.ssh/config`.
</Admonition>

The playbook below is a starting point that automates two tasks —
adapt it freely to match your own toolchain and workflow:

1. **Local**: adds the `vscode+` SSH config entry to `~/.ssh/config`.
2. **Remote**: connects to the Beam and installs Go plus common development
   packages, selecting the correct binary for the Beam's CPU architecture.

#### Inventory

Create `inventory.ini` pointing to your Beam:

```ini
[beams]
<beam-name>.<tenant-name>.beams.run ansible_user=beams
```

#### Playbook

Save this as `provision-beam.yml`:

{/* cspell:disable */}

```yaml
---
# Play 1: Add a VS Code SSH entry for every host in the [beams] inventory group.
- name: Configure local SSH for Beam VS Code access
  hosts: localhost
  connection: local

  tasks:
    - name: Add VS Code SSH config block
      ansible.builtin.blockinfile:
        path: "{{ ansible_facts['env']['HOME'] }}/.ssh/config"
        marker: "# {mark} ANSIBLE MANAGED BLOCK - Beam {{ item }}"
        insertbefore: BOF
        create: true
        mode: "0600"
        block: |
          Host vscode+{{ item }}
              HostName {{ item }}
              User beams
              UserKnownHostsFile /dev/null
              RemoteCommand bash
      loop: "{{ groups['beams'] }}"

# Play 2: Provision the Beam with Go and development tooling.
- name: Provision Beam with Go development tools
  hosts: beams
  gather_facts: true   # needed to detect CPU architecture

  vars:
    # "latest" resolves the current stable release from go.dev at run time.
    # Pin to a specific release with: -e go_version=1.26.4
    go_version: latest

  tasks:
    - name: Map Ansible architecture to Go architecture string
      ansible.builtin.set_fact:
        go_arch: "{{ 'arm64' if ansible_facts['architecture'] == 'aarch64' else 'amd64' }}"

    - name: Fetch latest stable Go version from go.dev
      when: go_version == "latest"
      ansible.builtin.uri:
        url: https://go.dev/dl/?mode=json
        return_content: true
      register: go_releases

    - name: Resolve latest Go version string
      when: go_version == "latest"
      ansible.builtin.set_fact:
        go_version: "{{ (go_releases.json | first).version | regex_replace('^go', '') }}"

    - name: Install common development packages
      ansible.builtin.apt:
        name:
          - git
          - make
          - build-essential
          - curl
          - jq
          - unzip
        state: present
        update_cache: true
      become: true

    - name: Check installed Go version
      ansible.builtin.command: /usr/local/go/bin/go version
      register: go_installed
      changed_when: false
      failed_when: false

    - name: Install Go {{ go_version }} ({{ go_arch }})
      when: >
        go_installed.rc != 0 or
        ('go' + go_version) not in go_installed.stdout
      block:
        - name: Download Go tarball
          ansible.builtin.get_url:
            url: "https://go.dev/dl/go{{ go_version }}.linux-{{ go_arch }}.tar.gz"
            dest: "/tmp/go{{ go_version }}.linux-{{ go_arch }}.tar.gz"
            mode: "0644"

        - name: Remove previous Go installation if present
          ansible.builtin.file:
            path: /usr/local/go
            state: absent
          become: true

        - name: Extract Go to /usr/local
          ansible.builtin.unarchive:
            src: "/tmp/go{{ go_version }}.linux-{{ go_arch }}.tar.gz"
            dest: /usr/local
            remote_src: true
          become: true

        - name: Add Go binaries to PATH in .profile
          ansible.builtin.lineinfile:
            path: "{{ ansible_facts['env']['HOME'] }}/.profile"
            line: "export PATH=$PATH:/usr/local/go/bin:$HOME/go/bin"
            create: true
```

{/* cspell:disable */}

#### Run

```code
$ ansible-playbook -i inventory.ini provision-beam.yml
```

To pin a specific Go version instead of auto-detecting the latest:

```code
$ ansible-playbook -i inventory.ini provision-beam.yml -e go_version=1.26.4
```

After the playbook finishes:

- `~/.ssh/config` on your local machine has the `vscode+<beam-name>.*` entry
  ready for VS Code.
- The Beam has Go and common packages installed.
