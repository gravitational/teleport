The tables below map Teleport's capabilities to specific [NIST SP 800-53 Rev 5](https://csrc.nist.gov/projects/cprt/catalog#/cprt/framework/version/SP_800_53_5_1_1/home) controls used by FedRAMP authorizations. Coverage applies uniformly across every access modality inside your authorization boundary — SSH, Kubernetes, databases, web applications, Windows desktops, and AI agents.

For deployment guidance (FIPS build, cluster configuration, agent setup) see the main [FedRAMP compliance page](./fedramp.mdx). For coverage against FedRAMP 20x Key Security Indicators, see the [FedRAMP 20x KSI mapping](./fedramp-20x.mdx).

The **Baseline** column shows the FedRAMP impact level at which each control applies — **L** (Low), **M** (Moderate), **H** (High) — per the [FedRAMP Rev 5 baseline profiles](https://github.com/OSCAL-Foundation/fedramp-resources/tree/main/baselines/rev5) (which tailor [NIST SP 800-53B](https://csrc.nist.gov/pubs/sp/800/53/b/upd1/final) for FedRAMP). Some controls apply only at Moderate and above, or only at High.

## Access controls

Teleport provides one access-control surface across all three identity classes inside your boundary — humans, workloads, and AI agents — with shared RBAC, audit, and lock primitives. Teleport contributes to the AC controls below through the same identity-aware proxy that mediates SSH, Kubernetes, database, and MCP tool invocations: every connection is authenticated against an SSO-sourced or platform-attested identity, authorized against role policy, and recorded in the audit log.

| Control | Baseline | Requirement | Teleport Features |
| - | - | - | - |
| [AC-02 Account Management]((=fedramp.control_url=)AC-02) | L, M, H | Define, document, and manage the lifecycle of accounts — request, approval, creation, modification, disabling, and removal. | Teleport integrates with SSO providers such as GitHub, Okta, Google, etc. Teleport supports role-based access control for accessing infrastructure. Certificate-based authentication and audit logging via Teleport. Audit events are emitted in the Auth Service when a user is created, updated, deleted, locked, or unlocked.<br /><br />**AI Agents:** Human-launched agents inherit lifecycle from the user who started them via a [delegated session](https://github.com/gravitational/teleport/blob/master/rfd/0238-delegating-access-to-ai-workloads.md). Autonomous agents are managed as [Bot resources](../../machine-workload-identity/machine-workload-identity.mdx) with explicit creation, role-scoped TTLs, and deprovisioning that emits audit events like any other identity. |
| [AC-03 Access Enforcement]((=fedramp.control_url=)AC-03) | L, M, H | Enforce approved authorizations for logical access to information and system resources in accordance with applicable access control policies. | Teleport Enterprise supports robust [Role-based Access Controls (RBAC)](../authentication/authentication.mdx) to:<br/>• Control which infrastructure resources a user can or cannot access.<br/>• Control cluster level configuration (session recording, configuration, etc.).<br/>• Control which Unix logins a user is allowed to use when logging into a server.<br /><br />**AI Agents:** Teleport's [MCP role spec](../../enroll-resources/mcp-access/rbac.mdx) lets you allow tool patterns like `slack_*` and deny specific tools like `slack_post_message` per agent identity. This enforces least privilege at every tool call — the same RBAC engine that scopes humans and workloads. |
| [AC-06 Least Privilege]((=fedramp.control_url=)AC-06) | M, H | Employ the principle of least privilege, allowing only authorized accesses for users (and processes acting on behalf of users) necessary to accomplish assigned tasks. | Teleport [RBAC](../../reference/access-controls/roles.mdx) supports fine-grained allow/deny rules per resource (servers, databases, Kubernetes clusters, MCP tools, web apps), letting you scope role definitions to exactly the access a task requires. [Access Requests](../../identity-governance/access-requests/access-requests.mdx) provide time-bound elevation when a user temporarily needs more, eliminating the "always-on admin" pattern.<br /><br />**AI Agents:** Each agent identity gets a role enumerating exactly the resources and tools it is allowed to use — no implicit privilege from network reachability, no scope drift from an accumulating set of API key permissions over the agent's lifetime. |
| [AC-07 Unsuccessful Logon Attempts]((=fedramp.control_url=)AC-07) | L, M, H | Limit consecutive invalid logon attempts and automatically lock the account when the limit is exceeded. | Teleport supports two types of users: local and SSO-based accounts (GitHub, Google Apps, Okta, etc). For local accounts, by default, Teleport locks accounts for 30 minutes after 5 failed login attempts. For SSO-based accounts, the number of invalid login attempts and lockout time period is controlled by the SSO provider. |
| [AC-08 System Use Notification]((=fedramp.control_url=)AC-08) | L, M, H | Display an organization-defined system use notification message or banner before granting access. | Teleport integrates with Linux Pluggable Authentication Modules (PAM). PAM modules can be used to display a custom message on login using a message of the day (MOTD) module within the Session management primitive. Pre-authentication MOTD is also available. |
| [AC-10 Concurrent Session Control]((=fedramp.control_url=)AC-10) | H | Limit the number of concurrent sessions for each account type or account to the organization-defined number. | Teleport administrators can define concurrent session limits using Teleport's RBAC. Supports both a maximum number of connections and maximum number of simultaneously connected users. |
| [AC-12 Session Termination]((=fedramp.control_url=)AC-12) | M, H | Automatically terminate user sessions after defined conditions or events requiring session termination occur. | Teleport can be configured to terminate active sessions as soon as the user's certificate expires. Users can exit a Teleport interactive session at any time by typing exit or sending an interrupt signal to the process for remote execution of a program. Administrators can apply [locks](../../identity-governance/locking.mdx) to forcibly terminate active sessions based on a variety of criteria.<br /><br />**AI Agents:** A compromised or misbehaving agent's identity can be [locked](../../identity-governance/locking.mdx) the same way a human's is — active sessions terminate immediately and new connections are blocked. Replaces the "rotate the API key and hope nothing breaks" pattern that takes minutes to propagate while the agent keeps running. |
| [AC-17 Remote Access]((=fedramp.control_url=)AC-17) | L, M, H | Establish and document usage restrictions and connection requirements for remote access; authorize remote access prior to allowing such connections. | Teleport administrators create users with configurable roles that can be used to allow or deny access to system resources.<br /><br />**AI Agents:** Agent access to internal resources flows through the same identity-aware proxy as human remote access — no direct network reachability, no out-of-band paths. Agent reach is bounded by role policy, not by network topology. |
| [AC-20 Use of External Systems]((=fedramp.control_url=)AC-20) | L, M, H | Establish terms and conditions consistent with trust relationships before authorizing use of external systems to access organizational information. | Teleport supports connecting multiple independent clusters using a feature called [Trusted Clusters](../management/trustedclusters.mdx). When allowing access from one cluster to another, roles are mapped according to a pre-defined relationship of the scope of access.<br /><br />**AI Agents:** An agent's access to external services — third-party APIs, external [MCP servers](../../enroll-resources/mcp-access/mcp-access.mdx), SaaS endpoints — is gated through Teleport when those services are enrolled, applying the same trust relationships, role policy, and audit controls used for human access to external systems. For broader unrestricted egress, combine with network-layer controls. |

## Audit and accountability

Teleport's audit log is the system of record for the access plane — every authentication, authorization decision, session, command, file transfer, and configuration change lands in a structured, append-only stream you can ship to your SIEM or hold in [tamper-evident storage](./fedramp-deployment.mdx#audit-storage-backend). Combined with replayable session recordings, this gives 3PAOs and incident responders a single query surface spanning SSH, Kubernetes, databases, web apps, desktops, and AI agents.

**AI Agents:** Everything below applies to agent actions uniformly — there are no new pipelines, no separate AI logging stack, no parallel SIEM integration to stand up. MCP tool invocations and session lifecycle emit [their own audit events](../../reference/audit-events.mdx) (codes TMCP001–TMCP006) into the same log, forwarded by the same SIEM connectors, governed by the same RBAC as human activity. Enrolling an MCP server into Teleport routes its activity through the same audit, RBAC, and SIEM path as the rest of the access plane.

| Control | Baseline | Requirement | Teleport Features |
| - | - | - | - |
| [AU-03 Content of Audit Records]((=fedramp.control_url=)AU-03), [AU-04 Audit Log Storage Capacity]((=fedramp.control_url=)AU-04), [AU-12 Audit Record Generation]((=fedramp.control_url=)AU-12) | L, M, H | Ensure audit records contain what occurred, when, where, source, outcome, and identity; allocate adequate storage; generate audit records at organization-defined system components. | Teleport contains an [Audit Log](../../reference/deployment/monitoring/audit.mdx) that records cluster-wide events such as:<br />• Failed login attempts.<br />• Commands that were executed (SSH "exec" commands).<br />• Ports that were forwarded.<br />• File transfers that were initiated.<br /><br />**AU-04 (Storage Capacity)** is supported by the [storage backend you configure](./fedramp-deployment.mdx#audit-storage-backend) — Teleport writes audit records to the backend, but capacity allocation and lifecycle management are customer-owned settings on the underlying store. |
| [AU-06 Audit Record Review, Analysis, and Reporting]((=fedramp.control_url=)AU-06) | L, M, H | Review and analyze system audit records for indications of inappropriate or unusual activity; report findings to designated personnel. | Teleport provides a unified, queryable audit log and session recordings that can be replayed for forensic review. [Access Monitoring](../../identity-governance/access-monitoring.mdx) surfaces unusual access patterns from the Teleport-side activity. **Teleport's contribution is access-plane evidence and replay** — broader SIEM correlation across non-Teleport systems, the review cadence, and the reporting workflow remain customer responsibilities. |
| [AU-10 Non-Repudiation]((=fedramp.control_url=)AU-10) | H | Provide evidence that protects against an individual or system falsely denying having performed an action. | Teleport audit logging supports both events as well as audit of an entire SSH session. For non-repudiation purposes, a full session can be replayed back and viewed. |
| [AU-11 Audit Record Retention]((=fedramp.control_url=)AU-11) | L, M, H | Retain audit records for an organization-defined time period consistent with records retention policy. | Audit log retention is governed by the [storage backend you choose](./fedramp-deployment.mdx#audit-storage-backend) — S3 with Object Lock for tamper-evident long-term retention, DynamoDB streams, etc. **Teleport doesn't enforce a retention period directly**; the backend's lifecycle policies do. Document your chosen retention period in your SSP. |

## Configuration management

Teleport emits audit events for every change to its own configuration — roles, users, cluster config, locks, bot resources — landing in the same audit log described in the Audit and Accountability section ([tamper-evident when configured with an append-only or WORM-style backend](./fedramp-deployment.mdx#audit-storage-backend)). Change reviewers and 3PAOs get the same "what changed, who, when" query surface they have for session activity.

**AI Agents:** Changes to the AI side of your environment — enrolling an [MCP server](../../enroll-resources/mcp-access/mcp-access.mdx), updating an agent's tool allow/deny list, locking an agent identity — emit the same configuration-change events as any other Teleport modification. The change-management evidence chain is the same one you're already running for humans and workloads.

| Control | Baseline | Requirement | Teleport Features |
| - | - | - | - |
| [CM-08 System Component Inventory]((=fedramp.control_url=)CM-08) | L, M, H | Develop and document an inventory of system components that accurately reflects the system. | Teleport maintains a live inventory of the access-plane resources enrolled in the cluster — SSH nodes, Kubernetes clusters, databases, web applications, MCP servers, and registered Bot identities — queryable by users (who see what they have access to) and administrators at any time. CM-08's full scope is broader than Teleport; combine this inventory with your other asset-tracking sources to cover non-Teleport components. |

## Identification and authentication

Teleport handles identification and authentication for every identity inside your boundary — humans, workloads, and AI agents — on a single platform. Humans authenticate via SSO with phishing-resistant MFA; workloads and agents receive short-lived cryptographic certificates issued by the Teleport CA, with no shared secrets or static credentials.

| Control | Baseline | Requirement | Teleport Features |
| - | - | - | - |
| [IA-02 Identification and Authentication (Organizational Users)]((=fedramp.control_url=)IA-02) | L, M, H | Uniquely identify and authenticate organizational users and processes acting on behalf of users. | Integrates with SSO providers such as GitHub, Okta, Google, etc. Acts as its own SSO provider. Supports the use of multi-factor authentication (MFA) including per-session MFA. Supports PIV-compatible hardware keys, as well as connection and user limits.<br /><br />**AI Agents:** Agents acting on behalf of a human run as [delegated sessions](https://github.com/gravitational/teleport/blob/master/rfd/0238-delegating-access-to-ai-workloads.md) tied to that user's authenticated, MFA-backed identity — the audit chain stays connected to the originating human. |
| [IA-03 Device Identification and Authentication]((=fedramp.control_url=)IA-03) | M, H | Uniquely identify and authenticate devices before establishing a connection. | Teleport requires valid x509 or SSH certificates issued by a Teleport Certificate Authority (CA) to establish a network connection for device-to-device network connection between Teleport components.<br /><br />**AI Agents:** Autonomous agents have their identity [attested by the cloud or platform they run on](../../reference/deployment/join-methods.mdx) at join time — the AWS account, Kubernetes cluster, CI runner, or TPM-equipped host cryptographically vouches for the device before Teleport issues a certificate. |
| [IA-04 Identifier Management]((=fedramp.control_url=)IA-04) | L, M, H | Manage system identifiers by assigning unique identifiers, preventing reuse, and disabling after a defined period of inactivity. | Maintains several unique identifiers: local users are required to be unique (unique username), roles have unique names and tied to organization roles via SSO, identifiers for devices are unique randomly generated IDs (UUID).<br /><br />**AI Agents:** Agent identities are managed as [Bot resources](../../machine-workload-identity/machine-workload-identity.mdx) with unique names; their certificates carry unique cryptographic identifiers (SPIFFE-compatible SVIDs) and short TTLs that prevent silent reuse. |
| [IA-05 Authenticator Management]((=fedramp.control_url=)IA-05) | L, M, H | Manage system authenticators by verifying identity prior to issuance, establishing minimum and maximum lifetime and reuse restrictions, and refreshing authenticators on a defined cadence. | Teleport's authenticators are short-lived X.509 and SSH certificates issued by the Teleport CA. Lifetime is configured per role (`max_session_ttl`); there are no default passwords to change, no shared secrets to manage, and certificates expire automatically. SSO authenticators (WebAuthn, hardware keys) are managed by the IdP under its own IA-05 controls.<br /><br />**AI Agents:** Agent authenticators are short-lived certificates rotated automatically by `tbot` before expiry. There is no static credential to refresh or rotate manually — the authenticator IS the cert lifecycle, anchored by a [delegated join method](../../reference/deployment/join-methods.mdx) at issuance. |
| [IA-08 Identification and Authentication (Non-Organizational Users)]((=fedramp.control_url=)IA-08) | L, M, H | Uniquely identify and authenticate non-organizational users or processes acting on behalf of non-organizational users. | Teleport's primary scope is organizational users and their workloads. For non-organizational users granted access through Teleport (contractors, federated partners, etc.), the same [SSO](../sso/sso.mdx), MFA, and PIV-compatible hardware key controls applied to organizational users (see IA-02) apply equally. Identification of broader non-organizational populations outside the Teleport-mediated path is typically handled by federation with your IdP or an external identity proofing service. |

## System and communications protection

Teleport is a private certificate authority for your environment, issuing short-lived X.509 and SSH certificates to every identity and using mTLS on every component-to-component and client-to-resource connection (with the [documented exceptions](./fedramp.mdx#mtls-exceptions) for health checks, web UI session establishment, OpenSSH compatibility mode, and the RDP leg from the desktop agent). In FIPS mode the cluster restricts negotiation to FIPS 140-validated cipher suites and CA keys can be backed by an HSM or KMS — the cryptographic posture the SC controls demand is the cluster's default operating mode, not an add-on.

**AI Agents:** Agent traffic uses the same cryptographic machinery — every connection from an agent through the Teleport Proxy to an [MCP server](../../enroll-resources/mcp-access/mcp-access.mdx), database, or Kubernetes API is mTLS, with certificates issued by the same CA on the same rotation cadence under the same FIPS controls. No separate TLS termination for AI workloads, no static API key over HTTPS, no parallel rotation to track.

| Control | Baseline | Requirement | Teleport Features |
| - | - | - | - |
| [SC-07 Boundary Protection]((=fedramp.control_url=)SC-07) | L, M, H | Monitor and control communications at the external and key internal boundaries; implement subnetworks for publicly accessible system components. | In the default reverse-tunnel deployment, Teleport eliminates inbound SSH, Kubernetes, database, and application ports on protected hosts — agents dial out to the Teleport Proxy and accept connections only over the established tunnel. This consolidates what would otherwise be a sprawl of per-service bastions, team-managed jump hosts, and bespoke ingress configurations into one identity-aware boundary point.<br /><br />**AI Agents:** The same consolidation extends to enrolled [MCP servers](../../enroll-resources/mcp-access/mcp-access.mdx) — the proxy is the single ingress for human, workload, and agent access. No per-MCP-server inbound paths to defend. |
| [SC-08 Transmission Confidentiality and Integrity]((=fedramp.control_url=)SC-08) | L, M, H | Protect the confidentiality and integrity of transmitted information. | All access traffic between Teleport components and between clients and target resources is protected by mTLS using certificates issued by the Teleport CA (with the [documented exceptions](./fedramp.mdx#mtls-exceptions) for health checks, web UI session establishment, OpenSSH compatibility mode, and the RDP leg). In FedRAMP/FIPS mode, only FIPS 140-validated cipher suites are negotiated. See SC-13 for the cryptographic algorithm enforcement details. |
| [SC-10 Network Disconnect]((=fedramp.control_url=)SC-10) | M, H | Terminate the network connection associated with a communications session at the end of the session or after a defined period of inactivity. | Teleport requires valid X.509 or SSH certificates issued by a Teleport Certificate Authority (CA) to establish a network connection for device-to-device network connection between Teleport components. Session termination is driven by certificate expiry, idle timeouts configured per role, and [identity locks](../../identity-governance/locking.mdx). |
| [SC-12 Cryptographic Key Establishment and Management]((=fedramp.control_url=)SC-12) | L, M, H | Establish and manage cryptographic keys when cryptography is employed within the system. | Teleport initializes cryptographic keys that act as a Certificate Authority (CA) to further issue x509 and SSH certificates. SSH and x509 user certificates that are issued are signed by the CA and are (by default) short-lived. SSH host certificates are also signed by the CA. Teleport supports Hardware Security Modules (HSM).<br/>Teleport Enterprise builds against a FIPS 140-validated library are available.<br/>In addition, when Teleport Enterprise is in FedRAMP/FIPS mode, Teleport will only start and use FIPS 140-validated cryptography. |
| [SC-13 Cryptographic Protection]((=fedramp.control_url=)SC-13) | L, M, H | Determine and implement organization-defined cryptographic uses; implement organization-defined types of cryptography. | Teleport Enterprise builds against a FIPS 140-validated library. In addition, when Teleport Enterprise is in FedRAMP/FIPS mode, Teleport will only start and use FIPS 140-validated cryptography. |
| [SC-17 Public Key Infrastructure Certificates]((=fedramp.control_url=)SC-17) | M, H | Issue public key certificates under an organization-defined certificate policy or obtain from an approved service provider. | Teleport initializes cryptographic keys that act as a Certificate Authority (CA) to further issue X.509 and SSH certificates. SSH and X.509 user certificates that are issued are signed by the CA and are (by default) short-lived. SSH host certificates are also signed by the CA. |
| [SC-23 Session Authenticity]((=fedramp.control_url=)SC-23) | M, H | Protect the authenticity of communications sessions. | Teleport SSH and TLS sessions are protected with SSH user and X.509 client certificates. For access to the Web UI, Teleport uses bearer token auth stored in a browser token to authenticate a session. Upon user logout, SSH and TLS certificates are deleted from disk and cookies are removed from the browser. |
