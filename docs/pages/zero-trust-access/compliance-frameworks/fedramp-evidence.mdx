When your 3PAO asks "show me how this works," Teleport supports concise, coherent evidence demonstrations. Instead of assembling evidence from a half-dozen tools, screenshots from the previous quarter, and policy documents that describe what *should* be happening, you share your screen with the assessor and walk through each control live. With practice, a single Teleport administrator can demonstrate and present evidence on short notice.

For the broader FedRAMP overview, see the [FedRAMP compliance page](./fedramp.mdx). For deployment guidance, see the [Deployment guide](./fedramp-deployment.mdx). For control-by-control coverage, see the [Rev 5 control mapping](./fedramp-rev5.mdx) or [20x KSI mapping](./fedramp-20x.mdx).

## Common assessor asks

The sections below are a runbook for common assessor asks during a live audit. Each one names why the question comes up, then walks through exactly what to do — written to be scanned, not read.

### "Prove MFA is enabled for login"

**When it comes up.** The AC and IA control families (and the IAM KSI theme) all require demonstrable MFA on every authentication path. Assessors want to see the prompt fire and as much of the MFA ceremony as they can see on a screenshare.

**What to do:**

1. Share your screen and open the Teleport Web UI sign-in page (or open a terminal and run `tsh login`).
2. Authenticate. The MFA prompt fires and initiates your IdP's [MFA flow](../sso/sso.mdx).

Optional:

- If using [per-session MFA](../authentication/per-session-mfa.mdx): once logged in, run `tsh ssh user@<a-protected-host>` to proceed through the next MFA check.

**Talking points:**

- *"Every login uses phishing-resistant MFA, and high-value sessions require an additional MFA tap on the connection itself."*

### "Ban an active user"

**When it comes up.** Session termination and "what happens when someone needs to be cut off *now*" sit in AC-12 and the IAM-06 KSI. Assessors love seeing this work live because it's the control most often claimed and rarely demonstrated.

**What to do:**

1. Have a colleague share their screen, log in, and start an SSH session through Teleport. Leave it open.
2. In your Web UI, open the [**Locks**](../../identity-governance/locking.mdx) page, or in another terminal run `tctl lock --user=<colleague> --message="Session terminated for FedRAMP demonstration"` — the `--message` text displays in the locked-out user's terminal at session drop.
3. Apply the lock. The colleague's terminal shows your message and the session terminates instantly.
4. Have the user attempt to log back in. Authentication fails until the lock is removed.
5. Remove the lock when the demo is done.

**Talking points:**

- *"Locks can target a user, role, MFA device, or host, and they take effect immediately across every Teleport-protected resource."*

### "Show me the secrets used to login"

**When it comes up.** Assessors want to know what authentication material exists on operator workstations — passwords, API tokens, SSH keys. The expected answer in a modern Teleport deployment is *almost nothing* beyond a short-lived TLS certificate.

**What to do:**

1. Open a terminal on your workstation.
2. Run `ls ~/.tsh/keys/` and walk through the structure with the assessor — per-proxy subdirectories under your username, no `id_rsa`, no shared secrets. Inside each proxy directory are short-lived TLS and SSH certificates plus the matching private key.
3. Run [`tsh status`](../../reference/cli/tsh.mdx#tsh-status). Point at the cluster, the proxy URL, the logged-in user, the active roles, and the remaining certificate lifetime (typically hours, not days).
4. Print the X.509 certificate in parsed form so the assessor can see and screenshot every field. The file lives at `<your-username>.crt` inside your proxy directory under the `tsh` keys directory (Teleport 17.0+; on earlier versions look for `<your-username>-x509.pem` in the same location). Use a read-only command-line viewer — no copies, no GUI tools that could lock or alter the file mid-session:

   <Tabs>
   <TabItem label="macOS / Linux">
   ```code
   openssl x509 -in ~/.tsh/keys/<proxy>/<your-username>.crt -text -noout
   ```
   </TabItem>
   <TabItem label="Windows">
   ```code
   certutil -dump %USERPROFILE%\.tsh\keys\<proxy>\<your-username>.crt
   ```

   `certutil` ships with Windows. It parses the PEM file in place and prints every field. The file is not modified.
   </TabItem>
   </Tabs>

5. Walk the assessor through the certificate fields in the output.

   - **Subject** — the Teleport username being asserted by the certificate.
   - **Issuer** — the Teleport cluster's internal CA. Proves the certificate was issued by your cluster, not anything external.
   - **Not Before / Not After** — the validity window. Read **Not After** aloud and compare to the current time. Typical lifetime is a few hours; this is the proof of "short-lived."
   - **Subject Alternative Name (SAN)** — includes the Teleport cluster name and the user's principal identifiers.
   - **Key Usage / Extended Key Usage** — restricted to client authentication; the certificate is not usable for arbitrary purposes.
   - **Custom extensions (Teleport-specific OIDs)** — encode the user's role membership, allowed logins, Kubernetes groups, and other authorization scope. Look for OIDs under the `1.3.9999` namespace. This is what makes the certificate a *carrier of authorization*, not just identity.



**Talking points:**

- *"There are no passwords or long-lived keys. The certificate expires automatically, the user logs in again, and the new certificate is bound to the same MFA-verified identity."*
- *"The certificate carries both the user's identity and their authorization scope — role membership, allowed logins, Kubernetes groups. It's signed by the cluster's internal CA, short-lived, and can be revoked instantly via a lock."*
- *"Note the private key sitting next to the certificate (`<your-username>.key` on Teleport 17.0+, or part of the legacy `-x509.pem` bundle on older versions) is also short-lived: generated fresh at every `tsh login` and rotated with the certificate. There is no persistent secret on this workstation that an attacker can steal and reuse beyond the certificate's lifetime."*

### "Show me a recorded user session"

**When it comes up.** Session recording is one of the highest-value evidence artifacts on the page — it touches AC-2, AU-10, the MLA KSI theme, and incident response readiness. Assessors usually want to watch a recording back end-to-end.

**What to do:**

1. In the Web UI, go to **Audit** → **Session Recordings**.
2. Pick a recent session with a few minutes of activity — the cleanest demo is an SSH or `kubectl exec` session.
3. Click play. The [recording](../../reference/architecture/session-recording.mdx) plays back in their browser with the terminal output and timing intact.
4. Explain what gets captured per resource type — they will likely ask, and the answer is not the same for every modality:

   - **SSH and Kubernetes `kubectl exec`** — the entire pseudo-terminal (PTY) output. The replay reproduces what the user saw. For tamper-resistant capture of the actual commands executed, enable [Enhanced Session Recording (BPF)](../../enroll-resources/server-access/guides/bpf-session-recording.mdx) on the host.
   - **Windows desktop** — the screen contents and mouse input. Keystrokes are not captured.
   - **Application access** — a stream of `app.session.request` audit events bucketed into 5-minute chunks. No video.
   - **Database access** — the executed queries and a stream of database audit events. No video.

5. If session recording is intentionally disabled anywhere in your environment (shared service accounts, legacy systems), be ready to point at the corresponding SSP language and explain the exception.

**Talking points:**

- *"Every modality has the appropriate recording: terminal replays for SSH and Kubernetes, screen capture for Windows desktops, and structured event streams for applications and databases. Everything lands in the same audit log with consistent metadata."*

### "Show me the audit logs"

**When it comes up.** The entire AU family of controls and the MLA KSI theme are about whether you can answer "who did what, when, from where, and was it authorized?" Audit logs are the answer.

**What to do:**

1. In the Web UI, go to **Audit** → **Audit Log**.
2. Filter for event types: `user.login`, `user.logout`, `session.start`, `session.end`, `lock.created`, `role.created`.
3. Click into one event to show the structured [JSON event](../../reference/deployment/monitoring/audit.mdx): user, timestamp, source IP, target resource, MFA method.
4. If your SIEM forwarding is configured, switch to the SIEM (Splunk, Panther, Elastic, Datadog) and run an equivalent query against the same events to demonstrate that the audit trail is being exported continuously, not just visible in the Teleport UI.

**Talking points:**

- *"Every access-plane action — logins, session starts and stops, role changes, lock events — is in this log, structured, queryable, and forwarded to our SIEM."*

### "Show me the maximum session length"

**When it comes up.** AC-12 and the IAM KSI theme both require that authenticated sessions don't live forever. Assessors will want to see the actual configuration value, not just hear that it exists.

**What to do:**

1. In the Web UI, go to **Access** → **Roles** (or run `tctl get role/<role-name>` in a terminal).
2. Pick a representative role. If you have differentiated populations (admins, contractors, read-only auditors), have a second role ready.
3. Surface the [`max_session_ttl`](../../reference/access-controls/roles.mdx) field. Read the value aloud.
4. Repeat for the second role to show the difference.

**Talking points:**

- *"Once the session TTL expires, the user has to re-authenticate from scratch — there's no silent renewal."*

### "Show me the inactivity timeout"

**When it comes up.** AC-11 (Session Lock) and AC-12 both require active sessions to terminate or lock on inactivity. The cleanest demonstration is to actually let it happen in front of them.

**What to do:**

1. On the same role you used above, surface the [`client_idle_timeout`](../../reference/access-controls/roles.mdx) field. Read the value.
2. Open a new `tsh ssh` session to a protected host. Type a command, get a response.
3. Leave the session idle. Keep narrating while you wait.
4. At the timeout boundary, the session disconnects automatically. Point at the closing message.

**Talking points:**

- *"The cluster enforces idle timeout on every active connection, not just at the application layer."*

### "Show me who can access what"

**When it comes up.** AC-2, AC-3, and AC-6 (Least Privilege) all hinge on whether your authorization model is documented, enforced, and reviewable. Assessors want to follow a user from IdP group → Teleport role → actual resource access.

**What to do:**

1. In the Web UI, go to **Access** → **[Roles](../../reference/access-controls/roles.mdx)**.
2. Pick a representative role. Walk through the `allow` block: which resources by label, which logins, which Kubernetes groups, which databases.
3. Open the role YAML (`tctl get role/<name>`) to show the same definition in source form.
4. Open your [SSO connector](../sso/sso.mdx) (`tctl get saml/<name>` or `tctl get oidc/<name>`) to show which IdP groups map to which Teleport role.

**Talking points:**

- *"The authorization model is plain YAML, version-controlled, and reviewable end-to-end. Every user inherits roles from their IdP group membership, and every role grants explicitly listed access."*
