This page maps Teleport's capabilities to the **FedRAMP 20x Key Security Indicators (KSIs)**. Coverage applies uniformly across every access modality inside your authorization boundary — SSH, Kubernetes, databases, web applications, Windows desktops, and AI agents.

For deployment guidance (FIPS build, cluster configuration, agent setup) see the main [FedRAMP compliance page](./fedramp.mdx). For coverage against the legacy NIST SP 800-53 Rev 5 control set, see the [Rev 5 control mapping](./fedramp-rev5.mdx).

<Admonition type="note" title="A moving target">

FedRAMP 20x is evolving rapidly. The KSI set below reflects **Phase Two (RFC-0014)** as of the "Last reviewed" date at the bottom of this page, integrating the retained Phase One KSIs. Phase Two ran from November 2025 through March 2026 and extends the program to Low **and** Moderate baselines. KSI IDs use the current named-suffix format (e.g., **KSI-IAM-MFA**) introduced in FedRAMP docs v0.9.0-beta on 2026-02-04; the former numeric IDs (KSI-IAM-01, etc.) are shown in italics for readers working from older artifacts. Always confirm current KSI text, phase status, and baseline applicability at [fedramp.gov/docs/20x/key-security-indicators/](https://www.fedramp.gov/docs/20x/key-security-indicators/).

</Admonition>

## How to read this page

Teleport's coverage of the 20x KSIs concentrates in four themes:

- **KSI-IAM** (Identity and Access Management) — broad, direct coverage
- **KSI-MLA** (Monitoring, Logging, and Auditing) — Teleport is your tamper-resistant access audit log
- **KSI-SVC** (Service Configuration) — cryptographic coverage for traffic and certificate lifecycle
- **KSI-CNA** (Cloud Native Architecture) — lateral movement, attack surface, traffic flow controls

KSIs outside these themes — cybersecurity education (CED), change management process (CMT), incident response process (INR), policy and inventory (PIY), recovery planning (RPL), and third-party risk (TPR) — are generally **customer responsibility** at the program level. Teleport's audit log and session recordings feed several of them (notably INR investigations and CMT change evidence), but the policies, processes, and broader platform controls live with your team.

Tables below show the KSIs where Teleport directly contributes to satisfying the controls. The **Class** column uses the FedRAMP Authorization Class designations **proposed** by [RFC-0020 / Notice-0004](https://www.fedramp.gov/notices/0004/), which would replace the legacy Low / Moderate / High naming. These designations are not yet final — FedRAMP plans to formalize them in the Consolidated Rules for 2026 (CR26), targeted for publication by the end of June 2026, with full transition to the class structure planned for January 2027.

| Class | Replaces | Notes |
| - | - | - |
| **A** | Pilot / Ready | New pilot baseline (Rev 5 only). No 20x path. |
| **B** | Li-SaaS and Low | 20x applies. |
| **C** | Moderate | 20x applies. |
| **D** | High | No 20x path. |

Until CR26 is published, the legacy Low / Moderate / High labels remain in force. The table below uses the proposed Class letters because the RFC and Notice make the mapping clear and these will be the operative terms by the time most readers are planning a 20x authorization — but if you're working an authorization today, expect to see Low / Moderate in the official artifacts. KSIs that vary by impact level (currently MLA-ALA and SVC-VCM) are noted as `B (opt), C` — required at Moderate, optional at Low.

## KSI-IAM — Identity and Access Management

Teleport provides one control surface across all three identity classes inside your boundary — humans, workloads, and AI agents — with shared RBAC, audit, and lock primitives. An agent's MCP tool invocation flows through the same controls as a human's SSH session.

| KSI | Class | Requirement | Teleport coverage |
| - | - | - | - |
| **KSI-IAM-MFA**<br />*(fka KSI-IAM-01)* | B, C | Enforce multi-factor authentication (MFA) using methods that are difficult to intercept or impersonate (phishing-resistant MFA) for all user authentication. | Teleport supports phishing-resistant MFA via WebAuthn, PIV-compatible hardware keys (YubiKey, Titan), and per-session MFA. SSO connectors enforce the IdP's MFA policy; local accounts can require MFA at login and per session.<br /><br />**AI Agents:** Agents launched by a human inherit that user's MFA-backed session through a [delegated session](https://github.com/gravitational/teleport/blob/master/rfd/0238-delegating-access-to-ai-workloads.md). Autonomous agents with no human in the loop have their identity [attested by the cloud or platform they run on](../../reference/deployment/join-methods.mdx) instead — see KSI-IAM-SNU for the mechanics. |
| **KSI-IAM-APM**<br />*(fka KSI-IAM-02)* | B, C | Use secure passwordless methods for user authentication and authorization when feasible, otherwise enforce strong passwords with MFA for authentication. | Teleport supports passwordless WebAuthn and hardware-key-bound login. When local accounts are used, MFA is required. In FedRAMP/FIPS mode, local authentication is disabled entirely — all access flows through your SSO provider.<br /><br />**AI Agents:** Agents are never given passwords — instead they authenticate with short-lived cryptographic certificates. See KSI-IAM-SNU for how the identity is bootstrapped. |
| **KSI-IAM-SNU**<br />*(fka KSI-IAM-03)* | B, C | Enforce appropriately secure authentication methods for non-user accounts and services. | Teleport's [Machine & Workload Identity](../../machine-workload-identity/machine-workload-identity.mdx) issues short-lived X.509 certificates and JWTs (SPIFFE-compatible SVIDs) to bots, CI/CD pipelines, and workloads. Static API keys and service-account passwords are replaced with auto-rotating cryptographic identities.<br /><br />**AI Agents:** Agents get their first credential through a [delegated join method](../../reference/deployment/join-methods.mdx) — the cloud account, Kubernetes cluster, CI runner, or TPM-equipped host they run on cryptographically attests to the agent's identity, and Teleport issues a short-lived cert in return. No static, long-lived secrets on disk, ever. |
| **KSI-IAM-JIT**<br />*(fka KSI-IAM-04)* | B, C | Use a least-privileged, role and attribute-based, and just-in-time security authorization model for all user and non-user accounts and services. | Teleport [RBAC](../../reference/access-controls/roles.mdx) supports role and attribute-based authorization with fine-grained allow/deny rules per resource. [Access Requests](../../identity-governance/access-requests/access-requests.mdx) provide just-in-time elevation with approval workflows and time-bound grants.<br /><br />**AI Agents:** Teleport's [MCP role spec](../../enroll-resources/mcp-access/rbac.mdx) lets you allow tool patterns like `slack_*` and deny specific tools like `slack_post_message` per agent identity. This enforces least privilege at every tool call. |
| **KSI-IAM-ELP**<br />*(fka KSI-IAM-05)* | B, C | Persistently ensure that identity and access management employs measures to ensure each user or device can only access the resources they need. | Every access decision is authenticated and authorized at the proxy on every connection. No standing network trust: hosts join via reverse-tunnel and serve only authenticated, certificate-bearing clients. Short-lived certificates limit blast radius if credentials are compromised.<br /><br />**AI Agents:** All agents are treated as [untrustworthy by default](../../machine-workload-identity/use-cases/ai-agents-mwi.mdx). You define policy for what the agent can do through Teleport; attempts that violate the policy are denied and audited. |
| **KSI-IAM-SUS**<br />*(fka KSI-IAM-06)* | B, C | Automatically disable or otherwise secure accounts with privileged access in response to suspicious activity. | [Session and identity locks](../../identity-governance/locking.mdx) can be applied to a user, role, MFA device, host, or Access Request — terminating active sessions and preventing new ones. Locks can be issued manually or automated via integration with SIEM/SOAR.<br /><br />**AI Agents:** A compromised or misbehaving agent's identity can be [locked](../../identity-governance/locking.mdx) the same way a human's is — active sessions terminate immediately and new connections are blocked. Replaces the "rotate the API key and hope nothing breaks" pattern that takes minutes to propagate while the agent keeps running. |
| **KSI-IAM-AAM**<br />*(fka KSI-IAM-07)* | B, C | Securely manage the lifecycle and privileges of all accounts, roles, and groups, using automation. | User lifecycle is sourced from the SSO IdP — provisioning, deprovisioning, and group/role changes propagate via SAML/OIDC. Local account changes emit audit events (create, update, delete, lock, unlock). [Access Lists](../../identity-governance/access-lists/access-lists.mdx) provide periodic reviews of role membership with required attestation.<br /><br />**AI Agents:** Human-launched agents inherit lifecycle from the user who started them — the agent's cert expires when the user's session does. Autonomous agents are managed as [Bot resources](../../machine-workload-identity/machine-workload-identity.mdx) with explicit creation, role-scoped TTLs, and deprovisioning that emits audit events like any other identity. |

## KSI-CNA — Cloud Native Architecture

Traditional network controls — VLAN segmentation, ACLs, jump hosts — leave gaps when applied alone to CNA-style restrictions in modern infrastructure. Teleport moves the controls to the identity layer: every connection from a human, workload, or AI agent terminates at an identity-aware proxy where it's authenticated, authorized against role policy, and logged. Lateral movement, attack surface, and traffic flow controls become enforced by the role policy rather than the (constantly shifting) network topology.

| KSI | Class | Requirement | Teleport coverage |
| - | - | - | - |
| **KSI-CNA-MAT**<br />*(fka KSI-CNA-02)* | B, C | Persistently ensure machine-based information resources have a minimal attack surface and that lateral movement is minimized if compromised. | In the default reverse-tunnel deployment mode, Teleport agents on protected hosts initiate outbound connections to the Teleport proxy — **no public inbound access is required** to those hosts. All access flows through the identity-aware proxy with per-connection authorization, eliminating the flat lateral-movement surface of traditional bastion or VPN designs. (Direct-mode agents do open local listening ports such as `3022`; reverse tunnels are the standard for production deployments and the recommended pattern inside a FedRAMP authorization boundary.)<br /><br />**AI Agents:** Teleport is the single access point for [every enrolled MCP server](../../enroll-resources/mcp-access/dynamic-registration.mdx) in your environment, replacing ad-hoc MCP sprawl with one managed, audited fleet. The same RBAC that scopes humans and workloads then limits each agent's reach inside your environment — bounded by role, not by network topology. |
| **KSI-CNA-ULN**<br />*(fka KSI-CNA-03)* | B, C | Use logical networking and related capabilities to enforce traffic flow controls. | Teleport enforces logical access-layer flow controls through its identity-aware proxy. RBAC rules define which users and workloads can reach which resources; [Trusted Clusters](../management/trustedclusters.mdx) extend the model across network boundaries with role mapping.<br /><br />**AI Agents:** Network ACLs can stop an agent from reaching a service; [MCP tool RBAC](../../enroll-resources/mcp-access/rbac.mdx) goes further by also constraining *what* the agent can do once it gets there. A single role expresses both which services an agent can act on and which actions it can take on each. |
| **KSI-CNA-DFP**<br />*(fka KSI-CNA-04)* | B, C | Strictly define the functionality and privileges for infrastructure and services. | Teleport eliminates long-lived credentials: users and workloads receive short-lived certificates scoped to their role, with no persistent secrets on hosts. Combined with `tbot` for machines, this aligns with immutable-infrastructure patterns — instances are deployed with a join token and identities expire on rotation.<br /><br />**AI Agents:** Every agent starts with zero access — [policies are written as explicit allow rules](../../reference/access-controls/roles.mdx), permit by exception, deny by default. An agent's effective capability is exactly what its current role enumerates, not the accumulated set of scopes an API key has picked up over its lifetime. |

## KSI-MLA — Monitoring, Logging, and Auditing

Teleport's audit log is the system of record for the access plane — every authentication, authorization decision, session, command, file transfer, and configuration change lands in a structured, append-only stream you can ship to your SIEM or hold in tamper-resistant storage. Combined with replayable session recordings, this gives 3PAOs and incident responders a single query surface spanning SSH, Kubernetes, databases, web apps, desktops, and AI agents.

**AI Agents:** Everything below applies to agent actions uniformly — there are no new pipelines, no separate AI logging stack, no parallel SIEM integration to stand up. MCP tool invocations and session lifecycle emit [their own audit events](../../reference/audit-events.mdx) (codes TMCP001–TMCP006) into the same log, forwarded by the same SIEM connectors, governed by the same RBAC as human activity. Enrolling an MCP server into Teleport routes its activity through the same audit, RBAC, and SIEM path as the rest of the access plane.

| KSI | Class | Requirement | Teleport coverage |
| - | - | - | - |
| **KSI-MLA-OSM**<br />*(fka KSI-MLA-01)* | B, C | Operate a Security Information and Event Management (SIEM) or similar system(s) for centralized, tamper-resistant logging of events, activities, and changes. | Teleport's [Audit Log](../../reference/deployment/monitoring/audit.mdx) captures all access-plane events (logins, role changes, session starts, commands executed, files transferred, port-forwards) in a structured, append-only stream. Logs can be forwarded to your SIEM (Splunk, Elastic, Datadog, Panther, etc.) or stored in DynamoDB/Firestore/S3 with object lock for tamper resistance. |
| **KSI-MLA-RVL**<br />*(fka KSI-MLA-02)* | B, C | Persistently review and audit logs. | Teleport provides the unified access log that makes this practical — every privileged session across SSH, Kubernetes, databases, web apps, desktops, and AI tool calls is in one place, queryable by user, host, time, and event type. Session recordings can be replayed for retrospective review. |
| **KSI-MLA-LET**<br />*(fka KSI-MLA-07)* | B, C | Maintain a list of information resources and event types that will be logged, monitored, and audited, then do so. | The Teleport audit event reference is published and versioned; the set of audited events is exhaustive for the access plane and stable across releases (additions are documented in release notes). |
| **KSI-MLA-ALA**<br />*(fka KSI-MLA-08)* | B (opt), C | Use a least-privileged, role and attribute-based, and just-in-time access authorization model for access to log data based on organizationally defined data sensitivity. | Teleport RBAC governs which users can read the audit log and session recordings via standard `event` and `session` resource rules. `where`-clause filtering supports patterns like "session participants only," and Access Requests can gate elevated log access just-in-time. |

## KSI-SVC — Service Configuration

Teleport is a private certificate authority for your environment, issuing short-lived X.509 and SSH certificates to every identity and using mTLS on every component-to-component and client-to-resource connection (with the [documented exceptions](./fedramp.mdx#mtls-exceptions) for health checks, web UI session establishment, OpenSSH compatibility mode, and the RDP leg from the desktop agent). In FIPS mode the cluster restricts negotiation to FIPS 140-validated cipher suites and CA keys can be backed by an HSM or KMS — the cryptographic posture KSI-SVC demands is the cluster's default operating mode, not an add-on.

**AI Agents:** Agent traffic uses the same cryptographic machinery — every connection from an agent through the Teleport Proxy to an [MCP server](../../enroll-resources/mcp-access/mcp-access.mdx), database, or Kubernetes API is mTLS, with certificates issued by the same CA on the same rotation cadence under the same FIPS controls. No separate TLS termination for AI workloads, no static API key over HTTPS, no parallel rotation to track.

| KSI | Class | Requirement | Teleport coverage |
| - | - | - | - |
| **KSI-SVC-SNT**<br />*(fka KSI-SVC-02)* | B, C | Encrypt or otherwise secure network traffic. | All access traffic between Teleport components and between clients and target resources is protected by mTLS using certificates issued by the Teleport CA. In FedRAMP/FIPS mode, only FIPS 140-validated cipher suites are negotiated. |
| **KSI-SVC-VRI**<br />*(fka KSI-SVC-05)* | B, C | Use cryptographic methods to validate the integrity of machine-based information resources. | Hosts and clients present X.509 / SSH certificates signed by the Teleport CA on every connection. Identity, role, and validity are cryptographically verified before any session is established. Session recordings can be encrypted by Teleport before storage; S3 Object Lock on the audit bucket independently provides tamper-evident retention. |
| **KSI-SVC-ASM**<br />*(fka KSI-SVC-06)* | B, C | Automate management, protection, and regular rotation of digital keys, certificates, and other secrets. | Teleport operates as a private CA that issues short-lived (typically minutes to hours) certificates to users, hosts, and workloads. Rotation is automatic; certificate lifetime is configured per role. CA keys can be stored in an HSM or KMS. |
| **KSI-SVC-VCM**<br />*(fka KSI-SVC-09)* | B (opt), C | Persistently validate the authenticity and integrity of communications between machine-based information resources using automation. | mTLS with Teleport-CA-issued certificates is the default for every component-to-component and client-to-resource path. Identity and integrity are cryptographically validated on every connection. |

## KSI-CMT — Change Management

Teleport emits audit events for every change to its own configuration — roles, users, cluster config, locks, bot resources — landing in the same audit log described in KSI-MLA ([tamper-evident when configured with an append-only or WORM-style backend](./fedramp-deployment.mdx#audit-storage-backend)). Change reviewers and 3PAOs get the same "what changed, who, when" query surface they have for session activity.

**AI Agents:** Changes to the AI side of your environment — enrolling an [MCP server](../../enroll-resources/mcp-access/mcp-access.mdx), updating an agent's tool allow/deny list, locking an agent identity — emit the same configuration-change events as any other Teleport modification. The change-management evidence chain is the same one you're already running for humans and workloads.

| KSI | Class | Requirement | Teleport coverage |
| - | - | - | - |
| **KSI-CMT-LMC**<br />*(fka KSI-CMT-01)* | B, C | Log and monitor modifications to the cloud service offering. | Teleport emits audit events for all access-plane configuration changes (role create/update/delete, user lifecycle events, cluster configuration changes, lock applications). When the audit backend is [configured for tamper-evident retention](./fedramp-deployment.mdx#audit-storage-backend) (e.g., [S3 Object Lock](https://docs.aws.amazon.com/AmazonS3/latest/userguide/object-lock.html)), these events provide durable evidence of who changed what and when, feeding the broader change-management evidence chain. |
