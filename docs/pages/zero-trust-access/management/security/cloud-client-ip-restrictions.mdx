Client IP Restrictions restrict access to your Teleport Cloud cluster, allowing
traffic only from the network ranges (CIDR blocks) you specify. You manage the
allowlist with the `client_ip_restriction` resource, using `tctl`, the Teleport
Terraform provider, or the Teleport Web UI.

Managing the allowlist with `tctl` and the Teleport Terraform provider is
available starting with Teleport v19. The Web UI is available in earlier
versions, but does not yet support the draft mode and expiry safeguards
described in this guide.

This guide covers how to:

- Configure a Teleport role that can manage Client IP Restrictions.
- Set the list of CIDR blocks allowed to connect to your Teleport Cloud cluster.
- Test an allowlist in draft mode and enforce it with an expiry, so that a
  misconfigured allowlist cannot lock you out permanently.
- Check whether the restrictions are enforced.
- Disable the restrictions.

## How it works

`client_ip_restriction` is a singleton resource: each Teleport Cloud cluster has
exactly one, and its name is always `client-ip-restriction`. Its
`spec.allowed_cidrs` field holds the list of CIDR blocks permitted to connect to
the cluster. An empty list disables restrictions and allows all traffic.

Two more `spec` fields control how and when the allowlist is enforced:

- `mode` sets your intent. `enforced` (the default when `mode` is empty or
  unset) tells Teleport Cloud to enforce the allowlist. `draft` saves the
  allowlist without enforcing it. Use `draft` to review a new allowlist without
  affecting live traffic, or to keep a vetted allowlist on standby so you can
  enforce it the moment you need to — for example, to lock the cluster down to
  trusted networks during a security incident.
- `expires` is an optional [RFC 3339](https://www.rfc-editor.org/rfc/rfc3339)
  timestamp. When it passes, Teleport Cloud stops enforcing the allowlist and
  reverts `mode` to `draft`. Use it as a safety net so that an allowlist that
  locks you out reverts on its own instead of requiring a support ticket. An
  unset value never expires.

When the allowlist is enforced, Teleport Cloud applies it at the cluster's
ingress layer: only clients whose source IP falls within one of the allowed CIDR
blocks can reach the cluster. Changes take 5-20 minutes to propagate and
terminate existing connections that no longer match the allowlist. In `draft`
mode the allowlist is stored but never applied, and all traffic is allowed.

The resource's `status.state` field reflects the current enforcement state:

| State     | Meaning                                                                                                      |
| --------- | ------------------------------------------------------------------------------------------------------------ |
| `draft`   | The allowlist is saved but not enforced, because `mode` is `draft`. All IPs can reach the cluster.           |
| `pending` | The allowlist has been saved and is being propagated to the ingress layer. Enforcement is not yet in effect. |
| `active`  | The allowlist is fully enforced at the ingress layer. Only IPs matching the allowlist can reach the cluster. |

Access to read and modify the allowlist is governed by Teleport's RBAC system
through the `client_ip_restriction` resource kind, and every change is recorded
in the [audit log](../../../reference/audit-events.mdx).

## Step 1/2. Configure RBAC

To read and modify Client IP Restrictions, a Teleport user needs permission to
manage the `client_ip_restriction` resource. The preset `editor` role already
has this access; if the user has the `editor` role, skip this step.

Create a file called `client-ip-restriction-editor.yaml`:

```yaml
kind: role
version: v8
metadata:
  name: client-ip-restriction-editor
spec:
  allow:
    rules:
      - resources:
          - client_ip_restriction
        verbs:
          - list
          - read
          - create
          - update
          - delete
```

Use `tctl` to create the role:

```code
$ tctl create -f client-ip-restriction-editor.yaml
```

Assign the role to a user. Open the user resource in your editor, replacing
`alice` with the Teleport username:

```code
$ tctl edit users/alice
```

Add `client-ip-restriction-editor` to `spec.roles`, then save and close the
editor to apply the change:

```diff
  spec:
    roles:
      - access
+     - client-ip-restriction-editor
```

## Step 2/2. Manage the IP allowlist

<Admonition type="danger" title="Avoid locking yourself out">
  A misconfigured allowlist can block all access to your cluster. Before you
  enforce changes, make sure the allowlist includes the network you are
  currently connecting from, as well as any third-party services that need to
  reach your cluster (CI/CD systems, identity providers, and so on). Teleport
  does not add these ranges automatically.

  Rather than enforcing a new allowlist directly, stage it in `draft` mode and
  then enforce it with an `expires` safety timer, as described in
  [Test and roll out the allowlist safely](#test-and-roll-out-the-allowlist-safely).
  The examples below enforce the allowlist immediately.
</Admonition>

<Tabs>
<TabItem label="tctl">

Create a file called `client-ip-restriction.yaml`:

```yaml
kind: client_ip_restriction
version: v1
metadata:
  name: client-ip-restriction
spec:
  allowed_cidrs:
    - "203.0.113.0/24"
    - "198.51.100.10/32"
```

Replace the entries in `allowed_cidrs` with the CIDR blocks allowed to connect
to your cluster. `metadata.name` must be `client-ip-restriction`; the resource
is a singleton and no other name is accepted.

Apply the allowlist:

```code
$ tctl create client-ip-restriction.yaml
# client_ip_restriction has been created
```

If Client IP Restrictions already exist, `tctl create` reports an error. Use the
`-f` flag to overwrite the current allowlist:

```code
$ tctl create -f client-ip-restriction.yaml
# client_ip_restriction has been created
```

You can also edit the allowlist interactively. `tctl edit` opens the current
resource in your editor and applies your changes when you save and exit:

```code
$ tctl edit client_ip_restriction
```

</TabItem>
<TabItem label="Terraform">

Before you begin, set up the
[Teleport Terraform provider](../../../configuration/terraform-provider/terraform-provider.mdx).

Create a file called `client-ip-restriction.tf`:

```hcl
resource "teleport_client_ip_restriction" "main" {
  version = "v1"
  metadata = {
    name = "client-ip-restriction"
  }

  spec = {
    allowed_cidrs = [
      "203.0.113.0/24",
      "198.51.100.10/32",
    ]
  }
}
```

Replace the entries in `allowed_cidrs` with the CIDR blocks allowed to connect
to your cluster. `metadata.name` must be `client-ip-restriction`; the resource
is a singleton and no other name is accepted.

Apply the configuration:

```code
$ terraform plan
# ...
# Plan: 1 to add, 0 to change, 0 to destroy.

$ terraform apply
```

For the full list of supported fields, see the
[`teleport_client_ip_restriction` resource reference](../../../reference/infrastructure-as-code/terraform-provider/resources/client_ip_restriction.mdx).

</TabItem>
<TabItem label="Web UI">

Log in to your Teleport Cloud account. Open the user dropdown menu on the top
right of the navigation bar and select "Help & Support," then scroll down to the

If you do not see the IP Allowlist section, the feature has not been enabled for

## Test and roll out the allowlist safely

A misconfigured allowlist can lock you out of your cluster. To avoid this, roll
out changes in two stages — first `draft`, then `enforced` with an expiry — so
that a mistake reverts on its own instead of becoming a permanent lockout that
requires a support ticket. This workflow is available with `tctl` and the
Terraform provider; the Web UI does not support it yet.

### Stage the allowlist in draft mode

Set `mode: draft` to save the allowlist without enforcing it. Teleport Cloud
stores the CIDR blocks but continues to allow all traffic, so you can review the
list with no risk of losing access.

<Tabs>
<TabItem label="tctl">

```yaml
kind: client_ip_restriction
version: v1
metadata:
  name: client-ip-restriction
spec:
  mode: draft
  allowed_cidrs:
    - "203.0.113.0/24"
    - "198.51.100.10/32"
```

Apply it with `tctl create -f client-ip-restriction.yaml`, then read it back and
confirm that `status.state` is `draft`:

```code
$ tctl get client_ip_restriction
```

</TabItem>
<TabItem label="Terraform">

```hcl
resource "teleport_client_ip_restriction" "main" {
  version = "v1"
  metadata = {
    name = "client-ip-restriction"
  }

  spec = {
    mode = "draft"
    allowed_cidrs = [
      "203.0.113.0/24",
      "198.51.100.10/32",
    ]
  }
}
```

Run `terraform apply`, then read the resource back with
`tctl get client_ip_restriction` and confirm that `status.state` is `draft`.

</TabItem>
</Tabs>

While the state is `draft`, verify that `allowed_cidrs` includes the network you
are connecting from and every third-party service that must reach your cluster.

You do not have to enforce the allowlist right away. You can leave it in `draft`
indefinitely as a vetted, ready-to-go configuration, then enforce it the moment
you need to — for example, to restrict access to trusted networks during a
security incident — by setting `mode: enforced` as shown below.

### Enforce the allowlist with an expiry safety net

When the draft looks correct, set `mode: enforced` and an `expires` timestamp a
short time in the future. `expires` must be in UTC and use
[RFC 3339](https://www.rfc-editor.org/rfc/rfc3339) format (the trailing `Z`
denotes UTC), and it must be at least 20 minutes ahead. Teleport Cloud enforces
the allowlist, but if it locks you out, enforcement automatically reverts to
`draft` at the expiry time and access is restored.

Compute a UTC timestamp 30 minutes from now:

<Tabs>
<TabItem label="Linux">

```code
$ date -u -d '+30 minutes' '+%Y-%m-%dT%H:%M:%SZ'
# 2026-01-01T00:30:00Z
```

</TabItem>
<TabItem label="macOS">

```code
$ date -u -v+30M '+%Y-%m-%dT%H:%M:%SZ'
# 2026-01-01T00:30:00Z
```

</TabItem>
</Tabs>

Use the output as the `expires` value:

<Tabs>
<TabItem label="tctl">

```yaml
kind: client_ip_restriction
version: v1
metadata:
  name: client-ip-restriction
spec:
  mode: enforced
  # Use the UTC timestamp from the command above.
  expires: "2026-01-01T00:30:00Z"
  allowed_cidrs:
    - "203.0.113.0/24"
    - "198.51.100.10/32"
```

</TabItem>
<TabItem label="Terraform">

```hcl
resource "teleport_client_ip_restriction" "main" {
  version = "v1"
  metadata = {
    name = "client-ip-restriction"
  }

  spec = {
    mode = "enforced"
    # Use the UTC timestamp from the command above.
    expires = "2026-01-01T00:30:00Z"
    allowed_cidrs = [
      "203.0.113.0/24",
      "198.51.100.10/32",
    ]
  }
}
```

</TabItem>
</Tabs>

<Admonition type="warning" title="Each write replaces the whole resource">
  Every write replaces the entire resource, so when you edit the allowlist you
  must re-send a still-valid `expires` or clear it. An `expires` value you read
  earlier and send back unchanged is rejected once it is within 20 minutes of
  the current time.
</Admonition>

After you confirm that you still have access, keep the allowlist enforced by
removing `expires` (or extending it) and reapplying. With `expires` unset,
enforcement never reverts automatically:

```yaml
kind: client_ip_restriction
version: v1
metadata:
  name: client-ip-restriction
spec:
  mode: enforced
  allowed_cidrs:
    - "203.0.113.0/24"
    - "198.51.100.10/32"
```

If you never set `expires`, the allowlist is enforced immediately and stays
enforced until you change it — with no automatic safety net. This is why we
recommend testing in `draft` mode first.

## Check the enforcement status

Retrieve the resource to view the current allowlist and enforcement state:

```code
$ tctl get client_ip_restriction
```

Example output:

```yaml
kind: client_ip_restriction
version: v1
metadata:
  name: client-ip-restriction
spec:
  mode: enforced
  allowed_cidrs:
    - 203.0.113.0/24
    - 198.51.100.10/32
status:
  state: active
```

The allowlist is fully enforced when `status.state` is `active`. A `pending`
state means the change is still propagating to the ingress layer, which can take
up to 20 minutes. A `draft` state means the allowlist is saved but not enforced,
because `mode` is `draft` (or an `expires` time has passed and enforcement
reverted).

## Disable Client IP Restrictions

To allow traffic from all IP ranges again, either apply an allowlist with an
empty `allowed_cidrs` list or delete the resource:

```code
$ tctl rm client_ip_restriction
# client_ip_restriction has been deleted
```

With Terraform, remove the `teleport_client_ip_restriction` resource from your
configuration and run `terraform apply`.

To stop enforcing the allowlist while keeping the CIDR list saved for later, set
`mode: draft` instead of deleting the resource. All traffic is allowed while the
allowlist is in `draft` mode.

## Limitations

- **Third-party service ranges.** Teleport does not add third-party service
  ranges automatically. Add allow rules for any third-party service that needs
  to access your Teleport cluster, such as CI/CD systems and identity providers.
- **Network security.** The allowlist applies only to Teleport Cloud access. It
  does not replace your organization's network or firewall policies.
- **Sync time.** Changes to the allowlist take 5-20 minutes to fully sync to the
  ingress layer.

## FAQ

### How do I test an allowlist without risking a lockout?

Stage it in `draft` mode first, confirm the list is correct, then enforce it
with a short `expires` safety timer. See
[Test and roll out the allowlist safely](#test-and-roll-out-the-allowlist-safely).
This workflow requires `tctl` or the Terraform provider; the Web UI does not
support it yet.

### What is the difference between draft mode and disabling restrictions?

`draft` mode keeps your CIDR list saved but stops enforcing it, so you can
review or resume it later. Disabling restrictions (an empty `allowed_cidrs` list
or deleting the resource) discards the list. Both allow all traffic.

### How many CIDR blocks can I configure?

By default, up to 256 CIDR blocks. Contact your account executive or support to
increase the limit.

### Do you support a denylist?

Teleport Cloud Client IP Restrictions do not currently support a denylist.

### Can I manage Client IP Restrictions with the Kubernetes Operator?

No. The Kubernetes Operator does not support the `client_ip_restriction`
resource. Use `tctl`, the Terraform provider, or the Web UI instead.
