The **Event Handler** plugin exports Teleport audit events to a Fluentd
service. The plugin retrieves events in configurable batches, forwards each
event to Fluentd over mTLS, and persists the ID of each successfully sent event
to local storage. If the plugin crashes or restarts, it resumes from the last
confirmed event. By default, the plugin polls for new events every 5 seconds.

## Configuration methods

The Event Handler plugin accepts configuration from three sources:

- Command-line arguments.
- Environment variables prefixed with `FDFWD_`.
- TOML configuration files specified with the `--config` flag.

TOML keys correspond to command-line argument names. You can group Teleport and
Fluentd settings into `[teleport]` and `[fluentd]` sections within the TOML
file.


## Teleport connection settings

| CLI argument | Environment variable | Default | Description |
|---|---|---|---|
| `teleport-addr` | `FDFWD_TELEPORT_ADDR` | None | Teleport Auth Service or Proxy Service host and port. |
| `teleport-ca` | `FDFWD_TELEPORT_CA` | None | Path to the Teleport TLS CA certificate file. |
| `teleport-cert` | `FDFWD_TELEPORT_CERT` | None | Path to the Teleport TLS certificate file. |
| `teleport-key` | `FDFWD_TELEPORT_KEY` | None | Path to the Teleport TLS private key file. |
| `teleport-identity` | `FDFWD_TELEPORT_IDENTITY` | None | Path to a Teleport identity file. Mutually exclusive with `cert`/`key`. |
| `teleport-refresh-enabled` | `FDFWD_TELEPORT_REFRESH_ENABLED` | `false` | Reload the identity file from disk on an interval. |
| `teleport-refresh-interval` | `FDFWD_TELEPORT_REFRESH_INTERVAL` | `1m` | How often to reload the identity file. |

## Fluentd connection settings

| CLI argument | Environment variable | Default | Description |
|---|---|---|---|
| `fluentd-url` | `FDFWD_FLUENTD_URL` | None | Fluentd URL for forwarding audit events. |
| `fluentd-session-url` | `FDFWD_FLUENTD_SESSION_URL` | None | Base URL for session events. Appends `.<session-id>.log`. |
| `fluentd-ca` | `FDFWD_FLUENTD_CA` | None | Path to the Fluentd TLS CA certificate file. |
| `fluentd-cert` | `FDFWD_FLUENTD_CERT` | None | Path to the Fluentd TLS client certificate file. |
| `fluentd-key` | `FDFWD_FLUENTD_KEY` | None | Path to the Fluentd TLS client private key file. |
| `fluentd-max-connections` | `FDFWD_MAX_CONNECTIONS` | None | Maximum number of concurrent connections to Fluentd. |

## Event processing settings

| CLI argument | Environment variable | Default | Description |
|---|---|---|---|
| `batch` | `FDFWD_BATCH` | `20` | Number of events to fetch per batch. |
| `timeout` | `FDFWD_TIMEOUT` | `5s` | Timeout for polling the Teleport event API. |
| `types` | `FDFWD_TYPES` | `*` | Comma-separated list of event types to forward. |
|`skip-session-types` | FDFWD_SKIP_SESSION_TYPES | `print`, `desktop.recording` | Comma-separated list of session event types to skip | 
| `skip-event-types` | `FDFWD_SKIP_EVENT_TYPES` | None | Comma-separated list of session event types to skip. |
| `start-time` | `FDFWD_START_TIME` | `now` | Minimum event time in RFC 3339 format. |
| `concurrency` | `FDFWD_CONCURRENCY` | `5` | Number of concurrent sessions. |
| `window-size` | `FDFWD_WINDOW_SIZE` | None | Exit when last event is processed. | 
| `storage` | `FDFWD_STORAGE` | Required | Directory for persisting event processing state. |

## Operational settings

| CLI argument | Environment variable | Default | Description |
|---|---|---|---|
| `debug` | `FDFWD_DEBUG` | `false` | Enable debug-level logging. |
| `dry-run` | `FDFWD_DRY_RUN` | `false` | Simulate execution without connecting to Fluentd. Useful for testing configuration. |
| `exit-on-last-event` | `FDFWD_EXIT_ON_LAST_EVENT` | `false` | Exit when last event is processed. |

## User auto-locking settings

The Event Handler plugin can automatically lock Teleport users after repeated
failed authentication attempts.

| CLI argument | Environment variable | Default | Description |
|---|---|---|---|
| `lock-enabled` | `FDFWD_LOCKING_ENABLED` | `false` | Enables automatic user locking after repeated failed login attempts. |
| `lock-failed-attempts-count` | `FDFWD_LOCKING_FAILED_ATTEMPTS` | `3` | Number of failed authentication attempts that trigger a lock. |
| `lock-period` | `FDFWD_LOCKING_PERIOD` | `1m` | Time window used to count failed authentication attempts. |
| `lock-for` | `FDFWD_LOCKING_FOR` | `None` | Duration that the user remains locked. |

For example, to lock a user after `5` failed attempts within `1m` for `30m`,
set:

```
lock-enabled = true
lock-failed-attempts-count = 5
lock-period = "1m"
lock-for = "30m"
```

## mTLS certificate generation

The Event Handler plugin requires mTLS between the plugin and Fluentd. Server
key encryption is recommended. Client key encryption is not supported.

### OpenSSL configuration

Create an `ssl.conf` file with the following structure. Adjust the
`[server_cert]` section to match your Fluentd server hostname and IP address.

```ini
[req]
default_bits       = 4096
default_md         = sha256
prompt             = no
distinguished_name = req_distinguished_name

[req_distinguished_name]
CN = localhost

[v3_ca]
basicConstraints     = critical,CA:TRUE
keyUsage             = critical,keyCertSign,cRLSign
subjectKeyIdentifier = hash

[client_cert]
basicConstraints     = CA:FALSE
keyUsage             = critical,digitalSignature
extendedKeyUsage     = clientAuth

[server_cert]
basicConstraints     = CA:FALSE
keyUsage             = critical,digitalSignature
extendedKeyUsage     = serverAuth
subjectAltName       = DNS:localhost,IP:127.0.0.1

[crl_ext]
authorityKeyIdentifier = keyid:always

[ocsp]
basicConstraints     = CA:FALSE
keyUsage             = critical,digitalSignature
extendedKeyUsage     = critical,OCSPSigning
```

### Generate a certificate authority

```code
openssl genrsa -out ca.key 4096
chmod 444 ca.key
openssl req -config ssl.conf -key ca.key -new -x509 -days 7300 \
  -sha256 -extensions v3_ca -subj "/CN=ca" -out ca.crt
```

### Generate a server certificate

```code
openssl genrsa -aes256 -out server.key 4096
chmod 444 server.key
openssl req -config ssl.conf -subj "/CN=server" -key server.key \
  -new -out server.csr
openssl x509 -req -in server.csr -CA ca.crt -CAkey ca.key \
  -CAcreateserial -days 365 -out server.crt -extfile ssl.conf \
  -extensions server_cert
```

### Generate a client certificate

```code
openssl genrsa -out client.key 4096
chmod 444 client.key
openssl req -config ssl.conf -subj "/CN=client" -key client.key \
  -new -out client.csr
openssl x509 -req -in client.csr -CA ca.crt -CAkey ca.key \
  -CAcreateserial -days 365 -out client.crt -extfile ssl.conf \
  -extensions client_cert
```

Alternatively, run the following command to generate all certificates to the
`example/keys/` directory:

```code
PASS=12345678 KEYLEN=4096 make gen-example-mtls
```
