## Access Monitoring Rule Spec

The access monitoring rule spec


Example:

```yaml
subjects: 
  - "string"
  - "string"
  - "string"
states: 
  - "string"
  - "string"
  - "string"
condition: "string"
notification: # [...]
automatic_approval: # [...]
```

|Field Name|Description|Type|
|---|---|---|
|subjects|Subjects the rule operates on, can be a resource kind or a particular resource property.|[]string|
|states|States are the desired state which the monitoring rule is attempting to bring the subjects matching the condition to.|[]string|
|notification|Notification defines the plugin configuration for notifications if rule is triggered. Both notification and automatic_approval may be set within the same access_monitoring_rule. If both fields are set, the rule handler will manage notifications and automatic approvals for the same set of access events. Separate plugins may be used if both notifications and automatic_approvals is set.|[Notification](#notification)|
|condition|Condition is a predicate expression that operates on the specified subject resources, and determines whether the subject will be moved into desired state.|string|
|automatic_approval|Automatic_approval defines the plugin configuration for automatic approvals. Both notification and automatic_approval may be set within the same access_monitoring_rule. If both fields are set, the rule handler will manage notifications and automatic approvals for the same set of access events. Separate plugins may be used if both notifications and automatic_approvals is set.|[Automatic Approval](#automatic-approval)|
