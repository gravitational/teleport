# Audit Events Generation Design

## Goal

Automate audit event boilerplate so that adding a new resource-gen managed resource produces compilable event infrastructure out of the box, with only resource-specific field customization left to the developer.

## Background

The scaffold (`service.go`) references event types and constants that don't exist yet:
- `apievents.ThingCreate` / `ThingUpdate` / `ThingDelete` / `ThingGet` structs
- `libevents.ThingCreateEvent` / `ThingCreateCode` string constants

The developer must currently define all of these by hand across 5+ files. This is the last remaining reason generated code doesn't compile immediately.

## Design

### Proto config addition

Add `code_prefix` to `AuditConfig` in `resource.proto`:

```protobuf
message AuditConfig {
  optional bool emit_on_create = 1;
  optional bool emit_on_update = 2;
  optional bool emit_on_delete = 3;
  optional bool emit_on_get = 4;
  string code_prefix = 5;
}
```

Validation rules:
- If any `emit_on_*` is true, `code_prefix` must be non-empty
- `code_prefix` must be 2-4 uppercase ASCII characters (e.g. "CK")
- `code_prefix` must be unique across all resources (validated at generation time)

### Generated constants

Event type strings follow `resource.<kind>.{create,update,delete,get}`.

Event codes follow `<PREFIX><NUMBER>I`:
- Create = 001
- Update = 002
- Delete = 003
- Get = 004

### Scaffold: event proto messages

resource-gen scaffolds `api/proto/teleport/events/v1/{kind}.proto` (created once, never overwritten).

Each enabled audit operation gets a message with standard embeds:

```protobuf
syntax = "proto3";
package teleport.events.v1;
import "teleport/legacy/types/events/events.proto";

message CookieCreate {
  events.Metadata Metadata = 1 [(gogoproto.embed) = true, ...];
  events.Status Status = 2 [(gogoproto.embed) = true, ...];
  events.UserMetadata User = 3 [(gogoproto.embed) = true, ...];
  events.ConnectionMetadata Connection = 4 [(gogoproto.embed) = true, ...];
  events.ResourceMetadata resource = 5 [(gogoproto.embed) = true, ...];
  // TODO: add resource-specific fields
}
```

Only messages for enabled operations are generated. The scaffold includes a comment reminding the developer to add these messages to the OneOf in `events.proto`.

### Always-regenerated files (gathering-style)

Six cross-resource files, regenerated on every run:

| File | Contents |
|------|----------|
| `lib/events/api.gen.go` | Event type string constants (`CookieCreateEvent = "resource.cookie.create"`) |
| `lib/events/codes.gen.go` | Event code constants (`CookieCreateCode = "CK001I"`) |
| `lib/events/dynamic.gen.go` | Dynamic factory registry via `init()` — maps event type strings to empty struct instances |
| `lib/events/events_test.gen.go` | Test map entries via `init()` |
| `api/types/events/oneof.gen.go` | ToOneOf converter cases via `init()` |

Each file uses the `init()` registration pattern: a `RegisterGenerated*` function in the existing hand-written code accepts registrations from the generated `init()`. This is the same pattern used by auth, cache, local parser, and tctl registration.

All files get the standard `// Code generated by resource-gen; DO NOT EDIT.` header.

### What the developer still does manually

1. **Customize event proto fields** — add resource-specific fields to `api/proto/teleport/events/v1/{kind}.proto`
2. **Add OneOf entries to events.proto** — add 2-4 lines to the OneOf message in `api/proto/teleport/legacy/types/events/events.proto`
3. **Customize emit methods** — fill in `FIXME` markers in `service.go` with resource-specific field population

### What is fully automatic

- Event type string constants
- Event code constants
- Dynamic event factory registration
- Test event map registration
- OneOf Go converter registration
- Scaffold event proto with standard embeds
- Scaffold emit methods (already exists today)

### Updated new-resource guide

Step 2 (`make grpc/host`) now also generates the event proto scaffold and all event registration files.

Step 3 (try to compile) changes: the scaffold compiles immediately for the standard fields. Only `FIXME` markers remain for resource-specific customization.

Step 4b (customize service scaffold) is simplified: the audit event types and constants exist automatically. The developer only needs to add resource-specific fields to the event proto and populate them in the emit methods.

A new sub-step is added: "Add OneOf entries to events.proto" with the exact lines to add.
